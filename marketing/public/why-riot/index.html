Community infrastructure that travels with people

Most community software works only while its company, server, and internet connection remain available. Riot keeps the useful parts — publishing, coordination, shared tools, and community memory — on people's own devices.

One story, three depths: For communities For partners For builders

The promise

Information, tools, and memory that travel with the people

Riot lets a community's information, tools, and memory travel with its people, even when the internet or a central service is unavailable.

Most collaborative products bind their authoritative state to a company account, server, relay network, or hosted database. Riot uses Willow so community data can remain useful on participants' devices and can be carried over different available paths.

Three internet roles are worth keeping separate as you read: a public web gateway renders an exported view for ordinary web readers; an internet seed or sync server carries Willow updates between remote Riot devices; and a nearby transport carries Willow updates directly between local devices. The current prototype supports local creation, local persistence, tested local-network (Bonjour) exchange, and public web rendering from exports. The server-mediated sync path between remote Riot devices is architectural direction with work in progress — not an integrated current product capability.

Depth one · for communities and organizers

Your community should still work when the network doesn't

Think of a tenants union carrying alerts, meeting changes, rides, and checklists. Disaster responders carrying shelter maps and supply requests into disconnected areas. A mutual-aid network coordinating needs, offers, schedules, and knowledge. An independent publication keeping and re-sharing its material when its website is blocked or a gateway disappears. These are examples of what Riot is designed to enable — not claims of completed field validation for each kind of organization.

Keep working locally

Reading, writing, organizing, and preparing updates happen on the device. No server round-trip is required for your community's ordinary work.

Exchange without internet

Nearby devices can connect, review offered updates, and choose whether to add them. Current evidence is prototype local-network (Bonjour) testing; physical two-iPhone Bluetooth exchange remains unverified. Tested prototype

Publish a public web view

The reference gateway can render exported public community content for ordinary browsers. It is not yet a live sync server for Riot devices. Working prototype

Carry more than messages

A community can carry structured information and community-approved tools — checklists, wikis, events, decisions — not just a chat log.

Manage several communities

Creating, following, switching, archiving, restoring, and reopening communities are supported in the core. The Using Riot guide names which screens are present on each platform.

Keep human control

External updates are previewed before acceptance, and unapproved community tools do not execute. Nothing is accepted automatically.

Riot turns every participating device into part of the community's library, newsroom, toolbox, and distribution network.

Depth two · for partners, funders, and journalists

Infrastructure communities can possess, not merely access

The structural difference from a conventional platform is where continuity lives. A Riot community's Willow data carries its own continuity — it is not derived from a domain name or a hosted account. Participant devices provide shared custody of existing community information. Public gateways are replaceable views over exported public data. Communities choose their tools and editorial practices, and public Newswires can combine open publishing with transparent editorial actions rather than invisible ranking.

Community media

Media without one canonical publishing server. A blocked website is a lost window, not a lost archive — the readers' devices still hold the material.

Coordination through failure

Ordinary community infrastructure — boards, schedules, checklists — that remains locally useful during outages, shutdowns, or censorship.

Tools each community chooses

Organizers approve the tools their community carries. There is no central app store deciding for every community at once.

Shared foundations

The same substrate serves mutual aid, tenant organizing, clinics, cooperatives, disaster response, protests, and independent media — plus future continuity between field exchange and remote internet sync.

This section deliberately makes no claim of production readiness, completed security audits, guaranteed availability, completed remote-device sync, or confidentiality for the public Newswire. The status section lists exactly what is working and what is direction.

Depth three · for builders and protocol readers

Willow separates shared data from the network carrying it

Riot publishes your update into the community's Willow space. Willow is designed so the update is not permanently tied to one transport. Riot's current prototype carries updates locally; internet seed synchronization is the next path under development.

Offline and online do not need separate community databases. They can be different ways of carrying the same shared state.

Under the hood

Willow provides

  • independent namespaces;
  • subspaces, paths, timestamps, and arbitrary payloads;
  • deterministic joins for partial replicas;
  • a data model independent of any one network transport; and
  • optional Meadowcap capability and Willow synchronization protocols.

Riot adds

  • community and profile semantics;
  • community relationships and per-community data identities;
  • public Newswire records and editorial actions;
  • community-approved, content-addressed mini-apps;
  • app-scoped bridge access and hardened native execution;
  • preview, validation, and acceptance policy;
  • durable multi-community management;
  • transport integrations and public export rendering; and
  • native interfaces for people who should never need to learn Willow.

Willow itself does not define Riot profiles, communities, Newswires, moderation, mini-apps, gateways, or native sandbox behavior. Riot mini-apps do not each receive an independent Willow namespace: their current isolation comes from app-specific paths inside the selected community plus app-scoped native bridge policy — not per-app namespaces, cryptographic isolation, or automatic end-to-end encryption.

Privacy

Privacy through control, not secrecy

The public Newswire is intentionally semi-public and plaintext. Alerts, mutual aid requests, public reporting, and community publications are often meant to circulate. This page will not pretend that widely shared information is secret.

Riot is privacy-respecting, not secret by default. Public community updates are meant to circulate. Privacy comes from reducing centralized collection, keeping community data on participants' devices, supporting separate community identities, limiting what tools can access, and letting people exchange information without always exposing their activity to internet infrastructure.

Riot cannot promise anonymity, conceal public posts, or erase every copy after information has spread. Encrypted private groups are planned separately.

You control your participation and your local data — not every copy of public information once it has been shared.

Willow possibility vs. current Riot behavior

The protocol can express more than Riot currently ships
Willow can express or supportCurrent Riot public communities
End-to-end encryptable dataPlaintext by design
Meadowcap scoped read/write capabilitiesWrite authority is checked; no confidential public read boundary
Confidential interest-overlap syncNot the current public sync path
Logical destructive editingNo recall or secure erasure of copies already received

On Meadowcap specifically: write authority travels with an entry and can be checked. Read authority is policy enforced by conforming replication peers. Read capabilities do not encrypt stored data by themselves; a malicious or authorized recipient can retain or re-share copied plaintext. Riot's public Newswire does not use Meadowcap as a confidentiality boundary.

What Riot does not hide

  • public Newswire content;
  • IP addresses or ordinary metadata visible to any internet service used;
  • nearby radio presence, device labels, proximity, and timing;
  • copies already carried away by another device, export, backup, or log; or
  • identity correlations caused by reused names, writing style, behavior, timing, proximity, or network metadata.

Separate per-community keys provide pseudonym separation at the data layer, not an anonymity guarantee. Community mini-apps receive app-scoped bridge access and run behind strong native network restrictions; their data is not secret, and zero egress is not absolute — the current Apple host documents a residual WebRTC hardening boundary.

The public web reader's trust boundary

Riot clients can independently validate canonical community data. The current public gateway renders a proof-free export for an ordinary browser. A browser reader therefore trusts the selected gateway's presentation — and may receive a view that is censored, stale, incomplete, fabricated, or incorrectly rendered. A gateway may also log connection metadata, like any website. "Replaceable" and "not the protocol authority" do not mean "cannot observe readers" or "cannot mislead a browser visitor."

Status

Working in the prototype / direction being built

Every material claim above carries its label where it first appears. This block is the summary, not a correction.

Working in the prototype

  • Durable core support for creating, following, switching, archiving, restoring, and reopening multiple communities
  • Native iOS community chooser and link/QR join flows
  • Local Newswire creation and durable display
  • Human confirmation and preview-before-accept import in nearby flows
  • Tested local-network (Bonjour) peer exchange
  • Public gateway rendering from exported community data
  • Community-approved mini-app packages
  • App-scoped bridge access and hardened app execution
  • Fresh per-community author identities when joining another community

Direction being built or still unverified

  • Encrypted private groups
  • Confidentiality for public communities
  • Full deletion from devices that already copied public content
  • Integrated remote Riot-device synchronization through an internet seed / anchor data path
  • Production scale, audited security, or guaranteed availability
  • Full interoperability with every Willow draft and transport
  • Physical two-iPhone Bluetooth exchange
